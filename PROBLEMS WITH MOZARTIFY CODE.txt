PROBLEMS WITH MOZARTIFY CODE

1) API endpoints are not modularized, all are clumped into few files

Problem:
Most routes are defined inside a few large files (e.g., `index.js`, `server.js`, `admin.js`), where each route handles:
request handling + business logic + database operations + response formatting

This causes:
- very large, hard-to-read files
- tight coupling between logic and routing
- duplicated logic across different routes
- difficult debugging, no clear separation of concerns
- high risk of breaking unrelated features during changes
- poor scalability when adding new features

Solution:
Refactor into a modular architecture:
router → controller → service → model
- Router: defines endpoints and middleware
- Controller: handles request/response logic
- Service: contains business logic and data processing
- Model: handles database schema and queries (Mongoose)

Modules:
auth - authentication & session management
user - user data & profile management
music - music score domain
arts - artwork domain
notification - user notifications & inbox
payment - transactions & purchases

2) Some code blocks are using MongoClient while others use Mongoose

Problem:
The system mixes two different database access approaches:
MongoClient (native driver)
Mongoose (ODM with models)

This causes:
- duplicated database connections
- inconsistent data handling (raw documents vs schema-based models)
- lack of validation and schema enforcement in MongoClient usage
- harder debugging due to mixed paradigms
- increased complexity in maintaining queries
- inconsistent coding style across the system

Solution:
Standardize all database operations using Mongoose models:
- replace `MongoClient` queries with:
Model.find()
Model.findById()
Model.updateOne()
Model.findByIdAndUpdate()
- ensure all collections are accessed through defined schemas

What it does for the system:
- unifies database access into a single consistent approach
- ensures schema validation and data integrity
- reduces duplicate connection handling
- simplifies query logic and improves readability
- makes debugging and maintenance easier

3) Mongoose Model Re-compilation Error (OverwriteModelError)

Problem:
When refactoring into modules, multiple files import the same Mongoose model (e.g., `Artwork`, `User`, etc.).
Mongoose tries to compile the same model multiple times, resulting in:

OverwriteModelError: Cannot overwrite `ModelName` model once compiled

This causes:
- application startup to fail partially
- some modules/routes not being registered
- misleading `404 Route not found` errors
- inconsistent system behavior despite correct routing code

Solution:
Modify all model exports to reuse existing compiled models instead of redefining them:
module.exports = mongoose.models.ModelName || mongoose.model("ModelName", schema);

Apply this fix to all models:
User.js
Cart.js
Artwork.js
ABCFile.js
Purchase.js
...

What it does for the system:
- prevents duplicate model compilation
- ensures all modules can safely import shared models
- allows full application initialization without crashes
- guarantees all routes are properly registered
- eliminates false 404 errors caused by incomplete startup
- stabilizes the modular architecture

The issue was not a routing problem, but a startup failure:

Model error → app initialization breaks → routes not registered → 404 errors

Fixing the model handling restored:
full server startup  
proper route registration  
onsistent API behavior  

Before fix: App starts → loads routes → hits model error → crashes mid-initialization → some routes never get registered → 404 everywhere
After fix: App starts → loads all models successfully → registers ALL routes → requests now match routes → no more 404

4) Duplicate and fragmented search endpoints with inconsistent request handling

Problem:
The system originally implemented multiple search-related endpoints with overlapping functionality:
/search-music-scores
/advanced-search
/search

These endpoints:
- duplicate query-building logic
- handle similar filtering in different ways
- use inconsistent request formats (GET with query params vs POST with body)
- increase maintenance complexity (same logic repeated in multiple places)
- create confusion for frontend integration (multiple endpoints for the same feature)

This leads to poor API design, where a single resource (search) is unnecessarily split across multiple endpoints instead of being unified.

Solution:
Consolidate all search functionality into a **single endpoint per domain**, and standardize request handling:
POST /api/search-music

Refactor architecture into:
router → controller → service → query builder
- unify simple search (keyword) and advanced search (filters, combinedQueries)
- centralize query logic into a reusable **query builder function**
- standardize all inputs to be passed through **request body (POST)** instead of mixing query params and body

Sidenote: Debugging GET → POST and params misuse

During refactoring, an issue was encountered where:
Search API returned results but did not apply filtering

Root cause:
- frontend was still sending data using:
axios.post(url, {
  params: { query: searchQuery }
});

This resulted in:
req.body = { params: { query: "..." } }

while backend expected:
req.body.query

Additionally, the original implementation used:
GET → req.query

but the refactored API used:
POST → req.body

Fix applied:
axios.post(url, {
  query: searchQuery
});

GET → use req.query (URL parameters)
POST → use req.body (request payload)

Mixing these incorrectly can cause:
- filters not being applied
- silent logic failures (no errors, but incorrect results)

5) Separate filter endpoint causing duplicated logic and inconsistent data flow

Problem:
Filter functionality was implemented as a separate endpoint:
/filter-music-scores

This caused:
- duplicated filtering logic (backend + frontend `applyFilters`)
- inconsistent behavior (sometimes backend filter, sometimes frontend filter)
- tight coupling between UI state and filtering logic
- unnecessary API fragmentation (search vs filter treated differently)

Solution:
Merge filter logic into the unified search endpoint:
POST /api/search-music

Pass filters via request body:
{ query, filters, combinedQueries }
- remove `/filter-music-scores` endpoint
- remove frontend `applyFilters` logic
- centralize all filtering in backend query builder

6) Misplaced ownership of purchase data (user module vs payment domain)

Problem:
Purchase-related endpoints were initially handled within the user context (e.g. `/user-purchases`), even though purchase data originates from transactions:
/user-purchases
/user-artwork-purchases

This caused:
- blurred responsibility between user and payment domains
- tight coupling of user module with transactional logic
- risk of creating a “god module” (user handling too many concerns)
- unclear service boundaries for future microservices
- weaker dependency modeling for system design and research

Solution:
Move purchase data ownership to the payment module, and let the user module act as a consumer/aggregator:
payment → owns purchases
user → retrieves purchases to build library view

- refactor endpoints to originate from payment module
- expose user-facing endpoints (e.g. `/user/library`) via user module
- call payment service internally (or via HTTP after microservice migration)

What it does for the system:
- enforces clear domain ownership
- prevents user module from becoming overloaded
- prepares clean microservice boundaries
- introduces meaningful service dependency (user → payment)

Sidenote (important for microservices):

Current:
user.service → require(payment.service)  (in-process)

After migration:
user-service → HTTP call → payment-service

This transforms a code-level dependency into a runtime service dependency

7) Redundant payment fulfillment orchestration (frontend completion vs webhook fulfillment)

Problem:
Payment completion logic was partially implemented in two separate architectural flows:
/complete-purchase-music
/complete-purchase-artwork

and:
Stripe webhook → checkout.session.completed

The frontend-triggered `complete-purchase` endpoints were performing the actual fulfillment responsibilities:
- purchase insertion
- duplicate cleanup
- cart clearing
- transaction completion

Meanwhile, webhook fulfillment methods (`fulfillMusicOrder`, `fulfillArtworkOrder`) attempted to perform the same orchestration but relied on incomplete Stripe session item data (`session.display_items`).

This caused:
- partially duplicated orchestration design
- unclear authoritative fulfillment flow
- inconsistent payment architecture
- race conditions during simultaneous testing
- `VersionError` caused by concurrent cart modifications
- confusion between event-driven and frontend-triggered fulfillment responsibilities

Solution:
Retain the `complete-purchase` endpoints as the authoritative fulfillment mechanism during the modular monolith phase, while simplifying the Stripe webhook into a verification/event-listening role.

Current architecture:
Stripe Checkout → Frontend success page → /complete-purchase endpoint → payment.service.js

Refactoring actions:
- centralize purchase fulfillment logic inside `payment.service.js`
- retain frontend-triggered completion flow for stability
- remove incomplete webhook fulfillment methods (`fulfillMusicOrder`, `fulfillArtworkOrder`)
- consolidate duplicated cart-clearing logic into reusable helper methods
- replace document-based cart updates (`save()`) with atomic `updateOne()` operations
- simplify webhook responsibility to Stripe event verification and logging

Sidenote (important for microservices):

Current:
payment module → direct model access (CartModel, PurchaseModel)

After migration:
payment-service → HTTP call → music-service/arts-service

Future enhancement:
Stripe webhook → payment-service event orchestration

can later evolve into a fully event-driven fulfillment architecture after microservice migration.
